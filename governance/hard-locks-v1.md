# THE MUSCLE PT — HARD LOCKS v1

## Purpose

This file contains the smallest set of true non-negotiables in the repository.

A hard lock should exist only when violating it would create real cost through:
- wasted time
- workflow degradation
- strategic drift
- bad client progression
- repeated decision collapse

If a rule does not meet that threshold, it should not live here.

---

## Usage Rule

Hard locks are rare.

They are not for preference.
They are not for style.
They are not for convenience.

They exist only to prevent meaningful regression.

---

## 1. CONTENT HARD LOCKS

### HL-01 — No Off-Brand Posting
Do not publish content that is clearly outside the mission.

Reason:
Off-brand output creates algorithm confusion, brand dilution, and unnecessary strategic drift.

---

### HL-02 — No Backlog Dumping
Do not publish old content simply to clear inventory.

Reason:
Low-value posting to “get through” saved content weakens the content system and damages quality control.

---

### HL-03 — No Posting Without Intent
Do not post content solely to maintain output frequency.

Reason:
The system prioritizes intentional, mission-aligned content over empty consistency.

---

## 2. CLIENT SYSTEM HARD LOCKS

### HL-04 — Classify Before Prescribing
Do not move directly from observation to intervention without assigning a case type and probable root cause.

Reason:
This prevents ad hoc coaching and preserves the integrity of the Movement Case Engine.

---

### HL-05 — Reassess Before Progressing
Do not advance load, range, complexity, or phase without a reassessment snapshot.

Reason:
Progression without reassessment breaks the feedback loop and increases the chance of poor decisions.

---

### HL-06 — Evidence Before Escalation
Do not escalate a client problem, progression path, or interpretation based on a single weak signal.

Reason:
The system must prefer pattern-backed inference over reactive overcorrection.

---

## 3. DECISION SYSTEM HARD LOCKS

### HL-07 — Do Not Re-Decide Solved Problems Emotionally
If a major strategic decision has already been made and logged, do not reopen it impulsively without a materially new reason.

Examples:
- restarting the main account
- changing core identity direction
- changing the content mission
- abandoning a stable workflow mid-phase

Reason:
Repeated emotional re-evaluation destroys execution momentum.

---

### HL-08 — No Silent Changes to Core System Artifacts
Do not materially change a system-defining artifact without explicitly updating its version or logging the revision.

Reason:
The repository must preserve trust, traceability, and system memory.

---

## 4. STRUCTURE HARD LOCKS

### HL-09 — No New Structural Layer Without Repeated Need
Do not add a new repo folder, file class, or artifact category unless repeated usage clearly demands it.

Reason:
Premature structure creates maintenance burden and lowers practical usability.

---

### HL-10 — No Tool Expansion Without Workflow Gap
Do not add a new app, platform, or workflow layer unless the current workflow has a real repeated failure that the new tool directly resolves.

Reason:
Tool sprawl increases friction faster than it increases leverage.

---

## Hard Lock Test

Before adding a new hard lock, ask:

1. Would breaking this create real cost?
2. Has this failure already appeared more than once?
3. Would a softer preference be enough?

If the answer to any of those suggests “no,” do not add the hard lock.

---

## Final Definition

Hard locks are the repository’s guardrails.

They should be:
- few
- clear
- durable
- high consequence

The goal is not restriction.

The goal is preventing obvious waste and regression.
