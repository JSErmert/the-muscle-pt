# THE MUSCLE PT — GOVERNING LOGIC v1

## Purpose

This document defines the governing logic of The Muscle PT system.

It unifies:
- execution (what gets done)
- governance (how decisions are constrained)
- architecture (how everything is structured)

The goal is to create a system that:
- executes quickly
- avoids repeated waste
- improves continuously
- scales without breaking

---

## Core Principle

Execute fast  
Constrain intelligently  
Escalate only when repeated  
Structure only when necessary

---

## 1. SYSTEM LAYERS

### 1.1 Execution Layer

Source:
Master Operating System

Controls:
- content production
- client sessions
- daily workflows
- weekly workflows

Priority:
Highest

Purpose:
Produce consistent output.

---

### 1.2 Governance Layer

Components:
- hard locks
- decision preferences
- escalation triggers

Controls:
- decision boundaries
- rule enforcement
- system stability

Purpose:
Prevent regression and guide decisions.

---

### 1.3 Architecture Layer

Controls:
- repository structure
- system organization
- versioning
- system relationships

Purpose:
Maintain long-term clarity and scalability.

---

## 2. EXECUTION PRIORITY RULE

Execution must never wait on governance unless a hard lock is triggered.

- Always act first.
- Use governance only to guide or correct.
- Do not delay output for system perfection.

---

## 3. DECISION HIERARCHY

All decisions follow this order:

1. Hard Locks  
2. Decision Preferences  
3. Execution Judgment  
4. Escalation (if repeated)

---

### 3.1 Hard Locks

- Prevent obvious waste
- Must not be violated
- Used rarely

---

### 3.2 Decision Preferences

- Guide direction
- Do not restrict execution
- Used frequently

---

### 3.3 Execution Judgment

- Used when rules do not apply
- Preserves speed and flexibility

---

### 3.4 Escalation

- Activated only when patterns repeat
- Converts friction into structure

---

## 4. EXECUTION FLOW

Execute  
→ Observe  
→ Reflect  
→ Escalate (only if repeated)

---

## 5. ESCALATION MODEL

Observation  
→ Repetition  
→ Pattern  
→ Escalation Trigger  
→ Promotion

---

### 5.1 Escalation Levels

Level 1 — Log  
Level 2 — Preference  
Level 3 — Hard Lock  
Level 4 — System Revision  

---

## 6. STRUCTURE GOVERNANCE

### 6.1 Minimal Structure Principle

Do not add structure unless repeated use demands it.

---

### 6.2 File Creation Rule

Create new structure only when:
- retrieval becomes difficult
- patterns repeat across artifacts
- separation improves clarity

---

### 6.3 Anti-Fragmentation Rule

- Do not split systems prematurely
- Prefer fewer, clearer files
- Avoid unnecessary hierarchy

---

## 7. RULE CREATION LOGIC

A rule should exist only if:
- a decision repeats
- a mistake causes real cost
- a pattern is stable

---

### 7.1 Promotion Rule

Repeated decision → preference  
Repeated cost → hard lock  
Repeated friction → system change
